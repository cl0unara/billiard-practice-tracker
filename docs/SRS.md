# Практична робота: Опрацювання вимог до програмного продукту

**Тема:** Billiard Practice Tracker (Система обліку тренувань з більярду)

## 1. Опис продукту
**Проблема:** Гравці в російський більярд часто тренуються інтуїтивно, не фіксуючи результати конкретних вправ (наприклад, "чужий у кут" або "свояк у середину"). Через це важко відстежити прогрес і зрозуміти, який тип удару «просідає».
**Рішення:** Billiard Practice Tracker — це консольний додаток на Python, який дозволяє гравцеві швидко записати назву вправи, кількість спроб і кількість забитих куль.
**Користь:** Система зберігає історію тренувань, що дозволяє гравцеві та його тренеру аналізувати ефективність тренувань у динаміці.

## 2. Стейкхолдери
1.  **Гравець (Користувач):** Головна особа, яка вносить дані.
    * *Інтерес:* Мінімальний час на введення даних, щоб не відволікатися від гри.
2.  **Тренер (або батько-наставник):** Людина, яка допомагає з технікою.
    * *Інтерес:* Перегляд статистики гравця для корекції плану тренувань.
3.  **Розробник (Студент):** Відповідає за реалізацію.
    * *Інтерес:* Вкластися в дедлайни навчання та використати знайомий стек (Python).

## 3. Функціональні вимоги (FR)
*Описують, що система повинна робити.*

* **FR-01:** Система повинна надавати можливість користувачу ввести назву тренувальної вправи.
* **FR-02:** Система повинна дозволяти вводити результат серії ударів (кількість забитих куль із загальної кількості спроб).
* **FR-03:** Система повинна валідувати введені дані (наприклад, кількість забитих куль не може бути більшою за кількість спроб).
* **FR-04:** Система повинна зберігати введені дані у локальний файл (формат .txt або .csv) після кожного запису.
* **FR-05:** Система повинна надавати можливість переглянути всю історію попередніх тренувань у хронологічному порядку.
* **FR-06:** Система повинна розраховувати та виводити відсоток успішності (ККД) для кожного збереженого запису.

## 4. Нефункціональні вимоги (NFR)
*Описують якість системи.*

* **NFR-01 (Продуктивність):** Час збереження нового запису у файл не повинен перевищувати 1 секунду.
* **NFR-02 (Сумісність):** Додаток повинен запускатися на будь-якій ОС (Windows, Linux, macOS), де встановлено інтерпретатор Python версії 3.10 або вище.
* **NFR-03 (Надійність):** У разі некоректного введення даних (літери замість цифр) програма не повинна "падати", а має вивести зрозуміле повідомлення про помилку.
* **NFR-04 (Зручність використання):** Інтерфейс має бути реалізований англійською мовою (латиницею) для уникнення проблем з кодуванням у консолі.
* **NFR-05 (Автономність):** Система не повинна потребувати підключення до Інтернету для виконання основних функцій.
* **NFR-06 (Доступність даних):** Формат файлу збереження має бути читабельним для людини (human-readable), щоб його можна було відкрити у звичайному Блокноті.

## 5. Первинний Backlog (User Stories)
*Список задач з пріоритетами (MoSCoW).*

**Епік 1: Введення даних**
* **US-01:** Як гравець, я хочу додати новий запис про вправу, щоб зафіксувати свій результат. (Priority: **Must**)
* **US-02:** Як користувач, я хочу отримувати попередження, якщо ввів текст замість числа, щоб випадково не зіпсувати дані. (Priority: **Should**)

**Епік 2: Перегляд та Аналіз**
* **US-03:** Як гравець, я хочу бачити список усіх своїх тренувань, щоб згадати, що я робив минулого тижня. (Priority: **Must**)
* **US-04:** Як тренер, я хочу бачити відсоток влучань біля кожної вправи, щоб швидко оцінити ефективність. (Priority: **Should**)
* **US-05:** Як гравець, я хочу мати можливість очистити історію, щоб почати новий сезон тренувань. (Priority: **Could**)
* **US-06:** Як користувач, я хочу експортувати дані в Excel, щоб будувати графіки. (Priority: **Won't** — на майбутнє).

## 6. Критерії приймання та Сценарії

### Критерії приймання для US-01 (Додавання запису)
* **AC 1.1:** Якщо користувач вводить назву вправи, кількість спроб "10" і влучань "7", система зберігає запис і виводить "Saved successfully".
* **AC 1.2:** Дані додаються у кінець файлу `history.txt`, не перезаписуючи старі записи.

### Основний сценарій використання
**Сценарій "Вечірнє тренування":**
1.  Користувач запускає файл `main.py`.
2.  Обирає опцію меню "1. Add new drill".
3.  Вводить назву: "Long pot" (Довгий чужий).
4.  Вводить: 10 спроб, 4 влучання.
5.  Система підтверджує запис.
6.  Користувач обирає "2. Show history", бачить свій запис і закриває програму.

### Definition of Done (DoD) для User Story
Історія вважається зробленою, якщо:
1.  Код написано і він відповідає вимогам PEP-8 (стандарт Python).
2.  Функціонал перевірено вручну (пройдено успішний сценарій).
3.  Код завантажено (push) у репозиторій GitHub.
4.  Немає критичних багів, що закривають програму.

### Деталізація NFR-03 (Надійність/Валідація)
* **Метрика:** 100% обробка виняткових ситуацій (Exceptions) при введенні типів даних `int`/`string`.
* **Спосіб перевірки:** Тестувальник навмисно вводить "abc" у поле для цифр.
* **Межа приймання:** Програма виводить "Error: please enter a number", а не системний текст помилки (Traceback).

### Ризики та їх зменшення
* **R-01 (Втрата даних):** Користувач може випадково видалити файл `history.txt`.
    * *Зменшення:* Програма автоматично створює новий файл, якщо старий не знайдено, щоб не впасти з помилкою.
* **R-02 (Некоректні дані):** Користувач введе, що забив 15 куль із 10 можливих.
    * *Зменшення:* Додати логічну перевірку `if scored > attempts: print("Error")`.
* **R-03 (Проблема кодування):** На Windows кирилиця може відображатися некоректно.
    * *Зменшення:* Використовувати тільки англійську мову в інтерфейсі (згідно з NFR-04).
